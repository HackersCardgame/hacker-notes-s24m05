Server side request forgery vulnerabilities are a seemingly benign attack vector that abuses the ability to invoke URL requests in web applications. Many websites will contain some form of invoking URL requests, the more common one's can be remote image uploads for avatars or web hooks to more esoteric means such as image and video uploads. The danger of this vulnerability manifests from the fact that a vast majority of internal services such as Redis, Memcached, Kubernetes or SMTP for example, can be attacked through a simple URL request which can then enable information disclosure from Kubernetes API keys to even sending legitimate emails from a victim's own SMTP server. This can also be leverage to access internal servers, metadata API's, and even access other connected devices to the internal network. Cloud services are quite vastly used such as Google Cloud, Amazon Web services, Azure and various services but many developers don't know about the inherent security risks that come from using them. Majority of cloud services provide developers a means to access metadata information which can include access tokens and credentials. This is normally accessible through metadata API which can be invoked from a local IP address. If an attacker can find a server side request forgery vulnerability in a target website, they can subsequently abuse the metadata API and gain full access to a victim's server. How a developer hardens against Server side request forgery involves input validation, compartmentalizing services, enforcing access control, as well as setting up security polices for cloud service metadata APIs. Although in some cases certain cloud services don't mention the dangers of these services and don't list ways to harden it on their end. For example, Microsoft Azure enforces passing an additional header to allow a request, and amazon web services allows enforcing where the call can originate from. In my case study I was able to successfully demonstrate to the U.S Department of defense how I was able to gain access to a sensitive internal network named NIRPNET or Non-classified Internet Protocol Router Network. As well as showing how i was able to abuse the meta-data API service of AWS to reveal sensitive instance based data. These attacks where disclosed through an on going vulnerability disclosure program which provides legal way for security researchers or hackers to disclose vulnerability and have them addressed. These are different from bug bounty programs which are exactly the same as vulnerability disclosure program but with teh caveat of being paid for a report as long as it followed the outline guidelines.
