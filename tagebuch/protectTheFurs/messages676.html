so, you want a decentralized consensus (amognus). this basically boils down to, given a past history H of blocks, choose some "canonical" next block B from candidates [B1, B2, B3...]. and everybody agrees on which one is actually the next block

option 1 - you ask some particular person / group of people (w/ a protocol like paxos or raft or whatever) to just choose the next block. but this is centralized, so.

option 2. you ask *everybody* to *vote* on their choice of next block. most votes wins. but who is everybody? this has issues with Sybil attacks - if an attacker just makes a few million "identities" and votes with 'em all, they can choose the next block arbitrarily. so that won't work.

option 2a. some authority issues individual humans "identities", signing a cert for 'em or something. this works... mostly... but is centralized. It also has a non-obvious problem: a single human can *vote for multiple candidates*- creating, say, a vote for B1 AND a vote for B2. and send those to different parts of the network. among other things this can cause big partition, etc. issues

option 2b. *work* "votes" instead of "identities". You prove you've done a lot of computation, you get one "vote". This also solves, if you want to make a currency, the problem of initially distributing resources or coins or whatever- if you reward doing that computation, to encourage folks to actually do it. Problem: this uses significantly more energy than most countries.

However, it also solves the "vote for multiple blocks" issue: an attacker would have to *split their work* up to do that- making a vote for B1 AND a vote for B2 takes twice as much work as making just one vote.

option 2c. *coins* vote instead of "identities"- one coin is one vote. *NOTE*: we have just introduced the idea of coins and economics and stuff into the core of our protocol! previously this was just a consensus mechanism... now we're needing to create an actual currency even if we just want consensus for some other thingy.

but there are problem. first: uhhhhh literally saying "one coin means one vote" is kinda, weird

second... you want to encourage people to vote, even when they don't particularly care about the outcome (e.g. in a cryptocurrency, even when they don't have any transactions in that block - (if they did have a tx in, say, B2 that wasn't in any other block, they'd naturally want to vote for that)) . why? because otherwise, relatively few people will vote... which means an attacker who wants to control consensus needs control of *way* less coins.

so you start paying people to "vote". ok. well, you still have the "vote for multiple blocks" issue, but now it's way worse- because everybody is incentivized to make a vote for *every block they see*. Our whole goal, remember, was "make sure there's only one 'canonical' chain, at least after a few more blocks have been added". if everybody votes for everything, that's the same as voting for *nothing*.

so, finally, 2d. this is 2c, but with two additions: you pay people to vote, *but you punish them heavily if they vote twice*. how can a decentralized network detect people voting twice? easy enough- if anybody can provide two votes signed by voter V for different blocks with the same parent, that proves voter V voted twice.
19:01
and you add to the protocol the rule that anybody who can provide two votes signed by the same voter - a proof that they 'cheated' - gets a small reward, and voter V gets a big punishment
19:02
now voters have an incentive to be honest and only vote once - especially if that punishment is big. and everybody has an incentive to catch and punish dishonest voters. that punishment is called "slashing".
19:04
but because a voter might just, say, move all their coins to a different address after making a dishonest vote, you have to make 'em lock up those coins for a while after each vote.
19:04
thus, proof of stake - where "staking" requires you lock up your coins for a while, keep a node online and make sure the software doesn't fuck up and accidentally vote twice (or you'll get slashed), and in return receive some reward.
19:05
now in practice it's not really "votes", it's more of "lottery tickets", but it's the same idea, just a monte carlo version of it
19:09

IF they also submit some proof of doing something else, which can be checked"

For example, Filecoin - "one coin gets one vote... IF they claim to be storing, say, 1TB of useful data, and submit proofs regularly that they still have that data". If they stop submitting proofs, they're punished (slashed).
19:10

https://margin.re/media/an-opinionated-guide-on-how-to-reverse-engineer-software-part-1.aspx is decent.

for me personally:

know some assembly, and in particular make sure you know the calling conventions for your platform.

when starting out, do it on binaries you can actually run in a debugger and see what's actually happening at runtime.

learn common patterns for things like "looping over an array", "iterating over a linked list", etc. this is very important.

the process, at least for me and most of the folks i've talked to, is one of iteration - start with what you know, use that to figure out other stuff. now you know more things, repeat.

So:

1. go for easy wins as much as you can. Strings! I love strings (mind you I don't do malware much). See a printf("access_check: ACL rule %d check denies action %s for user %s", foo, bar, baz)? Rename that function to access_check, rename foo to "acl_rule", rename bar to "acl_action", rename baz to "user". Now follow those back as far as you can, with everything that calls access_check. Something calls access_check(woof, bark, meow)? woof is acl_rule, bark is acl_action, meow is user. if you use Binary Ninja right it'll do a lot of this for you, not sure about other tools these days.

Look for logging functions especially - not only are they helpful, often they're called by macros that pass the function name as an argument - you can set the name of hundreds of functions with a script that way.

Also, symbols obviously help- not just exported/local symbols either. Use imports! if something's passed as the first argument to open, that's a filename.

2. structs structs structs structs. again, binary ninja does really well here because it propagates type information. but for most binaries I encounter, figuring out a handful of structs and looking at how they're passed between functions pretty much makes it clear exactly what's going on.

3. DON'T GET STUCK. if you can't figure out roughly what a function does (enough to give it a name) within a few minutes or so of looking at it, bookmark the function and move on- reverse other stuff. Very good chance that'll make it obvious what the function does. If not - remember why you are reversing this binary. If you're doing it to hunt for vulns... does the function handle user input or do security-critical things? If not, MOVE ON.

4. Cheat! see an interesting string? Google it, and search GitHub for it! You never know where open-source code pops up... (or where some helpful engineer has accidentally put proprietary code on GitHub). Not sure what protocol something's using? Look up the company's engineers on LinkedIn- what do they have experience with? Don't be afraid to use things other than the specific binary you're reversing to figure out how it works.
