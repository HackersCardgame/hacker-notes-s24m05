So... FPGA simulators, for verification purposes, *need* to know the behavior of the submodules in a design. Now, since Xilinx (and other companies) sell "IP cores" which are pre-packaged modules that people can use, to allow them to simulate it, they use a hack: They encrypt the source code with a symmetric algorithm (usually AES256), and then take its key, and encrypt it several times - each with the public RSA keys of the simulator vendors. This data is stuffed at the top of the file, before the encrypted blob (there's a whole IEEE standard for this; if you care, it's IEEE 1735-2014.)
Now, RSA has a fun property that I used: because of how it functions, the encrypted output of something MUST be (at a minimum) as long as the key used to encrypt it. This means, for a 1024-bit (128-byte) RSA key, if you encrypt 16 bytes, the output will be 128 bytes. How is this useful? Well, since the AES256 key is 256 bits, and we know it's base64'd from the metadata, we can look at the sizes of the different keys just by looking at the length of the encrypted data. This was how I noticed one key (Cadence's) had a 64-byte sized key.
