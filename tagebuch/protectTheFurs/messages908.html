.align(1) does mean that code will be aligned to the next modulo of 1 if not already aligned.

That allows you to control where in the memory layout a piece of code should be loaded which, back in the segmented DOS era, meant that you could add multiple pages of code and have them aligned by the loader without having to fiddle with filling the rest of page with zeroes.

It's also useful to ensure that you're memory-mapping stuff in the right place for a less-capable co-processor, which is required for older consoles that are now said to be retro, such as the PS2.
S
22:21
Shellspawn
Thanks for the question! @Sfner is correct, but it also forces the linker to not align the first section, which is important for making sure the first byte is start_external

The reason for the start_external is because the linker does not make any guarantees about how it will order the compiled functions. So if you just used _start and had other functions defined, it may put one of those as the first byte of the shellcode. start.s is specifically designed so the linker can't reorder it as it's in its own "execution unit".
